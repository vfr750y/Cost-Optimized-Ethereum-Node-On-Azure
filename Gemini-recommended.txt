Is there a way of achieving the same goal without using a VNET / Load balancer?

Yes, you can absolutely achieve this without a VNET or a Load Balancer by using a Public Azure Container Instance (ACI) Group.

In this configuration, the "firewall" logic shifts from the Azure Network Security Group (NSG) to the ACI Port Mapping itself. Since all containers in an ACI group share the same network stack and public IP, they can talk to each other on localhost, but only the ports you explicitly expose in the ACI definition are reachable from the internet.
How it works technically

    Lodestar Container: Listens on Port 9000 (P2P) and Port 8545 (RPC).

    Tailscale Container: Connects to your private Tailnet.

    ACI Configuration: You set the ip_address_type to Public and only include Port 9000 in the ports block of the ACI resource.

    Result: Azure will only route traffic from the internet to Port 9000. Port 8545 remains "invisible" to the public internet but is fully accessible to the Tailscale sidecar via localhost:8545.

As a system engineer, I have reviewed the updated documentation and investigated the technical requirements for Lodestar's peer discovery.
1. Fixed IP vs. Persistent URL

A fixed IP is NOT a hard requirement for a Lodestar light node, but a persistent URL (DNS name) is highly recommended if you are not using a static IP.

    How it works: Ethereum's discovery protocol (discv5) primarily uses IP addresses stored in ENRs (Ethereum Node Records). If your IP changes (common in ACI), your node's "reputation" with peers resets because they still look for you at the old IP.

    The Solution: By using the dns_name_label in Azure Container Instances, you get a persistent FQDN (e.g., my-node.eastus.azurecontainer.io).

    The Trade-off: While your node can still function, every time the ACI restarts and gets a new IP, it will take ~60 seconds to "re-announce" itself to the network. For a light client, this is acceptable.
